# utf8

[![CI](https://github.com/go-simd/utf8/actions/workflows/ci.yml/badge.svg)](https://github.com/go-simd/utf8/actions/workflows/ci.yml)
![coverage](https://img.shields.io/badge/coverage-100%25-brightgreen)

A drop-in fast path for UTF-8 validation and rune counting, byte-identical to
`unicode/utf8`. The bulk of the work runs a SIMD kernel generated by
[go-asmgen](https://github.com/go-asmgen/asmgen) (amd64 SSE/SSSE3 and AVX2); the
short tail and the rune that straddles the last SIMD block reuse the standard
library, so results match `unicode/utf8` exactly.

```go
ok := utf8.Valid(p)            // same bool as unicode/utf8.Valid(p)
ok := utf8.ValidString(s)      // same bool as unicode/utf8.ValidString(s)
n  := utf8.RuneCount(p)        // same int  as unicode/utf8.RuneCount(p)
n  := utf8.RuneCountInString(s) // same int as unicode/utf8.RuneCountInString(s)
```

| arch | kernel |
|---|---|
| amd64 | **SSE2/SSSE3 + SSE4.1** (16 B/block) and **AVX2** (32 B/block), runtime-dispatched |
| arm64 / loong64 / riscv64 | scalar (`unicode/utf8`) — NEON/LSX/RVV planned |

## Algorithm

### `Valid`

This is Lemire & Keiser's
[*Validating UTF-8 In Less Than One Instruction Per Byte*](https://arxiv.org/abs/2010.03090):
each byte is classified by its high nibble via `PSHUFB` nibble-lookups, then over
the `(prev, curr)` byte pair (`PALIGNR` / `VPERM2I128`+`VPALIGNR` to shift in the
previous block) a handful of checks run — continuation-length carry, the special
`ED`/`F4` first-continuation maxima, the overlong-encoding minima, and
`byte > 0xF4` — each ORing into a running error accumulator. The blocks are valid
iff the accumulator is all-zero (`PTEST`). Constant tables are emitted via
go-asmgen's `emit.File.Data`; the kernel is ported 1:1 from Lemire's reference
intrinsics. The caller backs the SIMD/scalar split up to a rune boundary so a
sequence straddling the last block (or an incomplete trailing sequence) is
re-validated in full by `unicode/utf8`.

Verified against `unicode/utf8.Valid` with a table test (ASCII, multibyte,
boundary/incomplete, and every invalid class — overlong, surrogate, too-large,
lone continuation), all lengths 0–300, and `FuzzValid`. Both the SSE and AVX2
kernels are fuzzed directly (~75 M executions, zero mismatches).

### `RuneCount`

The number of runes in **valid** UTF-8 equals the number of bytes that are *not*
continuation bytes — every encoded rune has exactly one leading byte, i.e. one
byte with `(b & 0xC0) != 0x80`. The kernel marks continuation bytes per block
(`[V]PAND 0xC0` then `[V]PCMPEQB 0x80`), extracts a bitmask (`[V]PMOVMSKB`) and
`POPCNT`s it, summing continuation bytes; the rune count is then `blockBytes −
continuations`.

That identity holds only for valid UTF-8: `unicode/utf8.RuneCount` treats each
byte of an erroneous or truncated sequence as its own 1-byte rune, so a
malformed run inflates the count beyond the non-continuation total. The kernel is
therefore only trusted over a block-aligned prefix the `Valid` SIMD validator
confirms is valid UTF-8; the rune straddling the last block and the tail are
counted by `unicode/utf8.RuneCount`, and **if the prefix is invalid the whole
input falls back to the scalar decoder**. The result is byte-identical to
`unicode/utf8.RuneCount` on *all* input, valid or not. (The SSE path uses
`POPCNT`, so it is gated on `cpu.X86.HasPOPCNT`; AVX2 CPUs always have it.)

`RuneCountInString` defers to `unicode/utf8.RuneCountInString`: feeding the SIMD
kernel a `string` would mean an allocating `[]byte` copy that dwarfs the saving
for the common short-string case.

Verified against `unicode/utf8.RuneCount` with the same table (ASCII, multibyte,
boundary/incomplete, and every invalid class), all lengths 0–300, `FuzzRuneCount`
and `FuzzRuneCountInString`, and a direct AVX2-kernel fuzz — all proven on a
**native AVX2** x86-64 host (~2 M executions, zero mismatches vs the standard
library).

## Performance

`Valid` throughput on a ~1 MiB mixed-UTF-8 buffer (ASCII-heavy with 2/3/4-byte
runes), **native amd64** (GitHub Actions `ubuntu-latest`, AMD EPYC 7763, AVX2,
`GOAMD64=v1`), `-count=6`, median MB/s. The dev box is arm64, where this
package's amd64 kernel only runs under Rosetta — and Rosetta hides AVX2 — so
these numbers come from CI on native hardware; see
[`.github/workflows/bench.yml`](.github/workflows/bench.yml).

## Prior art

- [`stuartcarnie/go-simd`](https://github.com/stuartcarnie/go-simd)
  (`unicode/utf8`) — the only pre-existing **pure-Go** SIMD UTF-8 validator: a
  2018 Go-assembly port of the same Lemire reference (SSE4 + AVX2, no cgo,
  Apache-2.0). It predates Go modules (no `go.mod`) and an explicit repo-root
  license, but go-gettable and benchmarked here as the competitor of record.
- [`charlievieth/simdutf`](https://github.com/charlievieth/simdutf) — a **cgo**
  wrapper around the C++ [simdutf](https://github.com/simdutf/simdutf) library;
  excluded from this pure-Go comparison.

## Coverage

The CI gate enforces **100% coverage of the Go code** on every arch job: native
amd64 (the Force tests drive the AVX2, SSE and POPCNT dispatch branches even on a
runner whose CPU otherwise always reports AVX2) and native arm64 (which covers
the `!amd64` generic fallback). Coverage is of the Go statements only: the
generated `.s` SIMD kernels are not measured by `go test -cover` — they are
validated by differential tests against `unicode/utf8` plus fuzzing (on a real
AVX2 box).
